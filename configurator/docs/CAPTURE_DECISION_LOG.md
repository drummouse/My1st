# IronWrap Capture — Decision Log

No project-wide decision-log file existed before Capture Stage 0; this file
is the canonical Capture decision log going forward. Format: one dated entry
per material decision, newest first. Reversing a decision gets a new entry —
old entries are never rewritten.

## 2026-07-19 — Stage 0 (repository audit)

| # | Decision | Rationale | Alternatives considered |
| --- | --- | --- | --- |
| D-001 | Build Capture inside the existing `configurator/` app as a bounded product area (new nav section + one consolidated `/api/capture` function), not a separate application. | Shares auth, tenancy, Neon, Blob, PWA, design tokens, tests, and deploys; discovery found no camera/offline constraint requiring a separate runtime. | Separate Vercel app (rejected: duplicated auth/session/domain logic, second deploy surface). |
| D-002 | Reuse Library Core (`library_records` + review lifecycle + versioning) as Capture's publication target; Capture never writes into `materials`/`colors` legacy tables or Studio project data. | Library Core already models `source_type='capture'`, `pending_review`, quality levels, versions, and audit — exactly the handoff contract in `CAPTURE_LIBRARY_HANDOFF.md`. Studio stays pinned because published records are new rows/versions, never mutations of consumed ones. | New parallel capture-product tables (rejected: Library fragmentation, double review systems). |
| D-003 | Extend the existing capability system (`superadminPolicy.js`) with `capture.create`, `capture.review`, `capture.publish.tenant` instead of a new role system. Owner gets all three scoped to its own tenant; superadmin gets them platform-wide; reseller gets none. | Tenancy is single-seat (one login = one company), so contributor and reviewer are the same account in MVP; capability names keep a future multi-seat split additive. Global publication remains superadmin-only (`catalog.publish`). | Dedicated contributor/reviewer roles (rejected for MVP: no multi-seat model to attach them to). |
| D-004 | Schema changes ship as strictly additive DDL through the existing runtime `ensureSchema()` + `db/schema.sql` mirror + parity tests. | It is the repo's only migration mechanism and is verified safe for previews: the Vercel↔Neon integration gives each git branch an isolated Neon branch (`preview/claude/development`), so merged DDL cannot touch production until a `main` deploy. | Introducing a migration tool now (deferred: valuable, but out of Capture's scope and already logged as platform debt in `ARCHITECTURE_AUDIT.md`). |
| D-005 | Capture audit events are written to `superadmin_audit_events` with `capture.*` action names. | Library Core already uses this table generically for `library.*` actions; one audit stream beats a parallel table. The misleading table name is accepted naming debt. | New `capture_audit_events` table (rejected: fragmenting audit). |
| D-006 | Server-side state machine states: `draft, submitted, in_review, changes_requested, approved, publishing, published, rejected, archived`. Sync-related spec states (`locally_queued`, `syncing`, `ready_to_submit`) are client/offline-layer states only; `resubmitted` is a flagged transition, not a state. | Fewer persisted states with identical auditability; offline states never round-trip to the server by design. | Persisting all 13 spec states (rejected: server would store client transport status). |
| D-007 | Uploads reuse `api/upload.js` Vercel Blob direct-upload with a new `capture` kind; originals immutable; derivatives are separate rows referencing `source_asset_id`; Neon stores URLs/metadata only, never Base64. | Existing, working, server-constrained direct upload path; keeps function bodies small and images out of Neon. | Uppy/tus (rejected for MVP: Blob doesn't support tus; ≤15 MB files don't need resumable chunking). |
| D-008 | Known gap accepted into Stage 2: Blob URLs are public-but-unguessable, not per-read authorized — same exposure as existing attachments/logos. Remedy decision (authenticated read-proxy vs provider signed reads) is deferred to Stage 2 and must be made before Capture handles genuinely sensitive imagery. | Matching current platform posture unblocks the MVP; the constraint is documented rather than silently violated. | Building a read-proxy now (deferred: costs the second-to-last Hobby function slot before it's needed). |
| D-009 | Stage 1 installs zero new dependencies. Full add/defer/reject table lives in the Stage 0 spec (`superpowers/specs/2026-07-19-capture-stage0-architecture.md`, §5). | Every MVP requirement in stages 1–3 is covered by installed packages plus native browser APIs. | — |
| D-010 | MVP measurements are stored as typed JSON in `capture_fields` (unit/method/confidence inside the value), not a dedicated `capture_measurements` table; processing-job tables deferred to Stage 7+. | Keeps the additive migration minimal for the first slices; a dedicated table becomes worth its rows when per-measurement provenance (AR/marker methods) arrives. | Full spec data model up front (rejected: broad schema before vertical slices). |
| D-011 | `00 - Project Vision` does not exist in the repository; recorded as absent per the master prompt. The July 5 single-file Configurator backup is also not in the repo and was not consulted as architecture. | Master-prompt rule compliance, recorded rather than invented. | — |
