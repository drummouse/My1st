# IronWrap Capture — Decision Log

No project-wide decision-log file existed before Capture Stage 0; this file
is the canonical Capture decision log going forward. Format: one dated entry
per material decision, newest first. Reversing a decision gets a new entry —
old entries are never rewritten.

| D-037 | R2 is implemented on `claude/ironwrap-capture-r2-adaptive-profile`, cut fresh from the verified `claude/development` baseline (`038394b2…`), replacing the earlier placeholder branch `claude/scanner-flexible-tags-a6kigl` (which carried zero commits beyond baseline). | D-038 | Local durability (`captureLocalStore.js`) uses an injectable storage-driver contract (`get/put/delete/getAll` per table) with two implementations: `createIndexedDbDriver` for real browsers, `createMemoryDriver` for unit tests — mirroring the injected-dependency pattern already used by `captureUploadQueue.js` (`performUpload`/`wait`). The only path that ever deletes local evidence is `confirmSynced`, called exclusively after a server-confirmed (committed) finalize response — never on a timer, never optimistically. Sync-state text (`saved_on_device`/`waiting_for_connection`/`uploading`/`upload_failed`/`synced`) is derived live from what's actually durable, never asserted from in-memory state. | Makes the §15/§17 "never display Saved when data exists only in temporary memory" rule true by construction instead of by convention, and keeps the queueing/rehydration/pruning logic fully unit-testable in Node without a browser or a fake-IndexedDB dependency. | A wrapper library (`idb`) (rejected: no new dependency needed for four methods); a single flat IndexedDB store instead of three (rejected: drafts/pending-assets/queue have different lifecycles and pruning rules). |
| D-039 | An accepted source image is never overwritten or deleted when "replaced." `capture_assets` gains one additive, nullable `superseded_by` column; `replaceAsset` inserts a fresh immutable asset (forced onto the SAME `purpose` as the one it replaces) and sets `superseded_by` on the prior row — the ONLY write ever made to that row, via a targeted `update capture_assets set superseded_by = … where id = …` that never touches url/checksum/capture_metadata/timestamps. `hasPhoto`/`hasView` (completeness + evidence) exclude superseded assets so only the current photo counts toward coverage. `removeAsset` (hard delete) remains for genuinely wrong shots pre-submission; `replaceAsset` is the path for "this view needs a better photo." | The R2 authorization is explicit: "never overwrite," "preserve timestamps/checksums/requested shot/acceptance history," "a simple retakeCount is not a substitute for lineage." Additive metadata/lineage is the sanctioned mechanism. | Deleting and re-inserting (rejected: destroys history, exactly what's prohibited); a separate `capture_asset_revisions` table (rejected: no current reader needs revision history beyond "what superseded what," which one FK-shaped column already answers). |
| D-040 | Deterministic quality/duplicate findings (sharpness, exposure/clipping, glare indication, crop/visibility sanity, perceptual-hash near-duplicate) are computed once, client-side, at accept time, and stored inside the SAME accepted asset's existing `capture_metadata` jsonb (`deterministicQuality: {pipelineVersion, findings}`, `perceptualHash`) — **no new `capture_deterministic_findings` table is created in R2.** Per the binding correction's own test: a dedicated table is justified when the design needs per-asset findings across *multiple* analysis passes, pipeline-version history, non-destructive re-analysis, reviewer inspection, or comparison of prior vs. current findings. R2 has exactly one pass per asset, computed once at accept time (the same lifecycle as `checksum`/`width`/`height`, already stored the same way), no re-analysis pipeline, and no reviewer UI that inspects a findings history — none of the table-justifying conditions hold yet. `pipelineVersion` is still recorded on every result so a future migration to a real table (when CV passes become repeatable/re-runnable) has a version to key off. | Honest, minimal-for-R2 engineering per the binding correction: "do not create a table only because it appeared in the planning document." Claude's analysis (R2.4) is the opposite case — repeatable, provenance-heavy, explicitly required to be a namespace separate from deterministic findings — and DOES get a dedicated table there, for exactly the reasons this table is deferred here. | `capture_deterministic_findings` table now, as the R2.0 plan proposed (rejected on inspection — see rationale); computing quality server-side (rejected: the accepted photo never needs to leave the device for this, keeps the check fast and offline-capable). |
| D-041 | Asset-level duplicate protection is checksum-based, session-scoped: `addAsset` treats a new SOURCE asset whose checksum matches an existing non-superseded source asset in the same session as a retry, returning the existing asset (`duplicate: true`, 200) instead of inserting a second row (fresh insert: `duplicate: false`, 201). Derived (thumbnail) assets are not deduped — a duplicate finalize skips thumbnail generation entirely rather than risk an orphaned extra derived row. | Closes the R2 gap: a flaky-network retry of the same accepted photo must not create two rows for one physical shot. Session-scoped (not global) because the same checksum in two different sessions is a legitimate coincidence, not a duplicate. | A separate idempotency-key header (rejected: checksum is already computed and already the natural identity of "this exact file"); a unique DB constraint on checksum (rejected: derived assets and cross-session legitimate repeats would violate it). |
| D-042 | `evaluateProfileEvidence()` (R2.3) surfaces the R2.2 deterministic quality findings additively as new `assetQuality`/`qualitySummary` fields. They are informational only: `phase`, `complete`, `shotRequests`, and `confidence` — the R1 contract — are computed exactly as before and are provably unaffected (a dedicated test evaluates identical asset/measurement input with and without quality findings attached and asserts every R1 field is byte-identical). Quality/duplicate findings never become a completeness gate, and never blend into the deterministic evidence `confidence` score — measurement/coverage confidence and quality findings stay two separate signals, matching "keep measurement confidence and geometry confidence separate." A superseded asset's findings drop out of `assetQuality` along with the rest of its coverage contribution (D-039). | Directly satisfies the R2 requirement to integrate deterministic findings into the existing evidence model "additively" without "replac[ing] existing R1 evidence contracts," while keeping the estimates honestly non-authoritative (§18). | Folding quality findings into `confidence` (rejected: conflates two different kinds of uncertainty — "is the view covered" vs. "is this specific photo any good" — into one number); gating `complete` on quality findings (rejected: these are heuristic estimates, not certified checks, and R1's calibrated-measurement authority must stay the actual gate). |
| D-043 | Claude vision privacy/data-flow decision recorded in full at `docs/CAPTURE_R2_CLAUDE_PRIVACY_DECISION.md` before implementing the live call, per the binding correction. Summary: Claude guidance reuses the EXISTING R2.2 on-device thumbnail (≤320px, already generated and stored as a `derived` capture_asset for every accepted photo) rather than inventing any new server-side image-processing step — the server fetches that thumbnail's existing Blob URL and sends it base64-encoded directly in the Anthropic Messages API request body. The original is never transmitted; no new temporary or public URL is ever created; assets without a thumbnail are simply excluded from the request rather than falling back to the original. Every attempt (success/disabled/unavailable/timeout/error/invalid) is recorded as an immutable row with non-sensitive diagnostic metadata; no image bytes are ever persisted in that record. Kill switch `CAPTURE_CLAUDE_GUIDANCE_ENABLED` defaults `false`; going live in a real environment additionally requires an owner-provisioned `ANTHROPIC_API_KEY` (never committed) — two independent gates, neither of which this implementation flips on. | Satisfies the binding correction's explicit list (which image, original-vs-derived, resolution policy, server-fetch-and-send vs. temporary URL, provider destination/retention, tenant disclosure, audit metadata, timeout/failure/deletion behavior, and whether full resolution is genuinely necessary) before writing any live-call code — and reusing the existing thumbnail keeps the zero-new-dependency rule intact (Vercel's Node serverless runtime has no built-in image resize capability; a new image-processing dependency would itself need a stop-and-ask). | Sending the original full-resolution photo (rejected: unnecessary — Claude is never authoritative for measurement, so full resolution serves no purpose §16 doesn't already forbid); generating a temporary signed URL for Claude to fetch (rejected: adds a new exposure surface for zero benefit over direct server-side fetch-and-send); a new server-side resize step via an image-processing dependency (rejected: the existing thumbnail already IS the right size — no new capability needed). |
| D-044 | Claude guidance gets its own dedicated table, `capture_claude_analyses` — the opposite call from D-040's deterministic-findings decision, and deliberately so: this table IS justified by the same criteria D-040 tests against. Every guidance attempt is its own row (repeatable — a session can ask again after capturing the recommended shot), versioned (`model`, `prompt_version`, `schema_version` per row), the record is immutable/append-only (no update path — a new attempt is a new row, not an edit), and the row is the exact separate-namespace home §16 requires ("stored separately from deterministic findings... user confirmations... reviewer decisions"). The row also carries `shot_request` (Claude's recommended next shot, if any) and `fulfilled_asset_id` (set once, compare-and-set, when that recommendation is captured) — no separate `capture_shot_requests` table, since in R2 there is at most one live Claude-authored shot recommendation per analysis and folding it into the same immutable row is simpler than a second table with its own lifecycle. | Directly satisfies "Immutable shot-request and analysis provenance" without over-building — one new table, not two. | A second `capture_shot_requests` table (rejected on inspection: no scenario in R2 needs a shot request to outlive or be queried independently of the analysis that produced it); updating a single row per session instead of appending (rejected: destroys the "what did we ask for and when" history the correction requires). |
| D-045 | Claude's response schema for R2 deliberately does NOT include a tags/applications suggestion field, even though the earlier Work Folder docs describe Claude "suggesting flexible tags" as part of its semantic-guidance role. | The binding correction excludes flexible-tag vocabulary/`item_type`/classification from R2 outright; rather than rely on a fine distinction between "advisory-only storage" and "persisting into a classification model," R2 avoids the topic entirely to remove any ambiguity. Revisit when the flexible-classification stage is actually authorized. | Including `suggestedTags` as an advisory-only, never-classification-linked field (rejected: unnecessary risk for a capability nothing in R2's vertical slice actually requires). |
| D-046 | R2.5's flat-wall preview is explicitly two layers, kept separate: (1) `evaluateFlatWallValidation()` in `captureStudioValidation.js` — pure, deterministic, unit-tested — decides readiness from confirmed measurements/material-zone/texture-direction only; (2) `CaptureFlatWallPreview.jsx` — an untested (matching the existing `CaptureCamera.jsx` canvas-glue precedent) client-side Three.js render of a plain wall plane plus a proportioned slab sized from those same real measurements, with simple color banding standing in for "texture direction" rather than any real texture map. Bends, seams, ribs, hems, and every other profile feature are never drawn — this is a scale/orientation/zone/direction proof, not geometry reconstruction. `capturePublish.js`'s Studio DTO `geometryUrl` is untouched (confirmed unchanged: still always `row.geometry_url ?? null`) — nothing from this preview is ever wired into it, and no GLB is produced or exported. Every UI/SVG label uses the required honest phrasing ("Technical compatibility preview," "Not reconstructed geometry," "Not fabrication grade"); the existing R1 SVG's aria-label was strengthened to match. | Directly satisfies the binding correction's flat-wall preview rules (parameterized placeholder / non-production Three.js test object only, must self-identify as schematic, must not be exported or published as Studio-ready) while keeping the testable logic (readiness) separate from the untestable rendering glue. | Rendering server-side for a "real" preview asset (rejected: no image-processing capability exists server-side per D-043, and R2 doesn't need a stored preview image — a live client render is sufficient for a technical proof); using the actual measured-profile SVG as a 3D extrusion source (rejected explicitly by the binding correction — the SVG is a 2D schematic, not valid extrudable geometry). |
